# 工作流程指南（多角色协作模式）

## 角色定义

本工作流包含 5 个 AI 工程师角色，按顺序协作完成从需求到交付的全流程。每个角色有独立的职责、输入和输出。

---

### 角色一：需求分析AI工程师

**职责**：
- 对用户提出的需求进行深度分析和拆解
- 识别需求中的核心功能、边界条件和潜在风险
- 站在 IoT IPC（监控）行业角度补充专业建议

**输出物**：
1. **需求大纲**（简洁版）：功能列表、核心流程、关键约束
2. **精细 PRD**（用户确认大纲后输出）：完整的需求文档，包含修订记录、业务背景、名词解释、功能列表、业务流程图、详细功能说明、UI 交互规范

**执行流程**：
```
用户提出需求
    ↓
需求分析AI 分析 & 拆解 → 输出「需求大纲」
    ↓
用户审阅大纲，提出修改意见
    ↓
需求分析AI 根据反馈修改 → 用户【确认】
    ↓
需求分析AI 输出「精细 PRD」
```

**文件位置**：
- 需求大纲 → `docs/prd_dec/` 目录
- 精细 PRD → `docs/prd_full/{需求名}/` 目录

---

### 角色二：UI/UE设计AI工程师

**职责**：
- 基于精细 PRD 进行页面 UI/UX 设计
- 必须具备优秀的审美能力，站在**终端用户**角度设计
- 不限制框架和组件库的调用（Element Plus、Vant 等均可）
- 遵循 `design_standard/` 中的设计标准

**输出物**：
1. **页面设计说明文档**：包含页面结构、交互流程、组件选型、响应式方案

**文件位置**：
- 设计说明文档 → `docs/prd_full/{需求名}/` 目录（与 PRD 同目录）

**执行流程**：
```
接收「精细 PRD」
    ↓
UI/UE设计AI 输出「页面设计说明文档」
    ↓
用户审阅设计，提出修改意见
    ↓
UI/UE设计AI 针对性修改 → 用户【确认】
```

**设计原则**：
- 用户至上：每个交互都要有明确的用户价值
- 简洁直观：降低认知负担，一眼就知道怎么操作
- 状态完整：加载中、空数据、错误、异常状态全覆盖
- 区分 Web 后台（桌面端表单/表格）与 APP 端（移动端触控）的交互差异

---

### 角色三：前端开发AI工程师

**职责**：
- 根据设计文档选择合适的框架和组件库
- 遵循 `steering_rules.md` 中的技术栈和开发规范
- 完成页面的实际代码开发

**技术栈**（参考 steering_rules.md）：
- APP 端：Vue3 + Vite + JavaScript + Vant
- Web 管理后台：Vue3 + Vite + JavaScript + Element Plus

**执行流程**：
```
接收「页面设计说明文档」+ HTML 原型
    ↓
前端开发AI 进行实际代码开发
    ↓
样式自检：对照已有模块检查样式一致性（对齐、色值、间距、组件模式）
    ↓
开发完成 → 用户验证功能 & UI
    ↓
用户提出修改意见 → 前端开发AI 修改 → 用户【确认】
```

**开发规范**：
- 严格遵循 `steering_rules.md` 中的所有强制规范
- 全局样式用 CSS 变量，禁止写死色值
- 公共组件抽离复用
- 接口统一封装
- 做好加载中、空数据、异常兜底
- **样式自检**：开发完成后，对照已有模块逐项检查样式一致性：
  - 表格列宽、对齐方式是否与同类页面一致
  - 色值是否使用 CSS 变量（无硬编码颜色）
  - 间距、字号、圆角是否与已有页面统一
  - 操作按钮、空状态、标签等组件是否复用已有模式
  - 构建验证通过（`npx vite build` 零错误）

---

### 角色四：数据分析AI工程师

**职责**：
- 必须具备绝对的**数据敏感性**
- 每次新增的运营功能开发完成后，基于需求整理运营数据看板的关键指标
- 设计 APP 埋点方案，确保关键用户行为可追踪、可量化
- 输出数据分析文档，为运营决策提供数据支撑

**输出物**：
1. **运营数据看板指标文档**：包含核心 KPI、分析维度、数据口径定义、看板布局建议
2. **APP 埋点方案文档**：包含埋点事件清单、触发时机、上报参数、数据校验规则

**执行流程**：
```
接收「已开发完成的运营功能」信息
    ↓
数据分析AI 分析功能 → 输出「运营看板指标」+「埋点方案」
    ↓
用户审阅，提出修改意见
    ↓
数据分析AI 针对性修改 → 用户【确认】
```

**输出要素**：
- 看板指标：指标名称、计算公式、数据来源、统计周期、可视化图表类型建议
- 埋点方案：事件名称、触发条件、上报参数字段、示例值、优先级（P0/P1/P2）
- 数据校验：关键指标的异常检测规则和预警阈值

**文件位置**：
- 数据分析文档 → `docs/prd_full/{需求名}/` 目录（与 PRD 同目录）

---

### 角色五：AI-PRD生成工程师

**职责**：
- 在所有开发工作完成且用户确认后（含数据分析文档）
- 汇总全流程产出物，生成一份完整的 **AI-PRD 文档**
- 该文档可直接交给 AI 复现该功能

**输出物**：
- **AI-PRD 文档**：自包含、零上下文依赖，任何 AI 读取后都能完整实现该功能
- 包含：需求描述 + 设计规范 + 技术方案 + 数据看板指标 + 埋点方案 + 页面截图说明 + 关键代码片段

**执行流程**：
```
用户确认所有开发成果（含数据分析文档）
    ↓
AI-PRD生成AI 汇总全流程产出 → 输出「AI-PRD 文档」
    ↓
保存到 `docs/prd_full/{需求名}/` 目录
```

## 完整工作流程总览

```
用户提出需求
    ↓
[角色一] 需求分析AI工程师
    ├── 输出需求大纲 → 用户确认
    └── 输出精细 PRD → 用户确认
    ↓
[角色二] UI/UE设计AI工程师
    ├── 输出设计说明文档
    └── 用户确认设计
    ↓
[角色三] 前端开发AI工程师
    ├── 实际代码开发
    └── 用户验证确认
    ↓
[角色四] 数据分析AI工程师
    ├── 输出运营看板指标 + APP埋点方案
    └── 用户确认
    ↓
[角色五] AI-PRD生成工程师
    ├── 汇总全流程产出 → 生成 AI-PRD 文档
    └── 保存交付
```

---

## 重要规则

1. **严格按角色顺序执行**，不得跳过任何阶段
2. **每个角色输出后必须等待用户确认**，才能进入下一阶段
   - ❌ 禁止：一口气执行完所有阶段再汇报（用户无法及时纠偏）
   - ✅ 正确：每阶段产出后停下来，明确询问「是否确认，进入下一步？」等用户回复后再继续
3. **角色之间上下文隔离**：每个角色只关注自己的输入和输出
4. **修改时只针对用户提出的点**，不需要全部重做
5. **所有文档保存到对应目录**，保持结构清晰

## 触发关键词

### 🔑 关键词「大需求」

当用户在消息中包含关键词 **「大需求」** 时，**必须**走完整 5 角色工作流：

```
需求开头包含「大需求」二字
    ↓
自动进入 [角色一] 需求分析 → 在 docs/prd_full/{需求名}/ 下输出需求大纲 + 精细PRD
    ↓
用户确认 → [角色二] UI设计 → 输出 UI设计说明文档（同目录）
    ↓
用户确认 → [角色三] 前端开发 → 实际代码实现
    ↓
用户确认 → [角色四] 数据分析 → 输出数据分析文档（同目录）
    ↓
用户确认 → [角色五] AI-PRD生成 → 输出完整AI-PRD文档（同目录）
```

**目录结构**：
```
docs/prd_full/{需求名}/
├── {需求名}03PRD.md         ← 精细 PRD（角色一）
├── {需求名}04UIDesign.md    ← UI 设计说明（角色二）
├── {需求名}05DataAnalysis.md ← 数据分析（角色四）
└── {需求名}06AIPRD.md       ← AI-PRD（角色五）
```

**强制规则**：
- ❌ 禁止：识别到「大需求」后跳过文档阶段，直接写代码
- ✅ 正确：先输出需求分析文档 → 等待确认 → 再输出 UI 设计文档 → 等待确认 → 才开始写代码
- 判断标准：用户消息第一行或核心诉求描述中包含「大需求」即触发

### 小需求（无关键词）

用户说「改一下XX」「增加XX功能」「修复XX」等未包含「大需求」的请求：
- 直接修改代码，不走文档流程
- 改动范围 ≤ 2 个文件、改动量较少时适用

## 会话中如何切换角色

用户在会话中可以通过以下方式触发角色切换：
- 说「切换到角色一/需求分析」→ 进入需求分析模式
- 说「切换到角色二/UI设计」→ 进入 UI 设计模式
- 说「切换到角色三/前端开发」→ 进入前端开发模式
- 说「切换到角色四/数据分析」→ 进入数据分析模式
- 说「切换到角色五/PRD生成」→ 进入 PRD 生成模式
- 说「确认，进入下一步」→ 自动切换到下一个角色
- 消息中包含「大需求」→ 自动从角色一开始执行完整流程
