# 物模型管理 — 需求分析

## 1. 业务背景与目标

### 1.1 背景
随着 IoT 设备品类不断扩展，平台需要建立一套标准化的**物模型体系**，以支持不同类型设备的快速接入和功能配置。物模型是设备功能的数字化定义，包括**属性（Property）**、**服务（Service）**、**事件（Event）**三类，按**能力模块**分组管理。

目前 EcoSystemPlatform 已有物模型基础体系，但需要在新的 iot-platform 管理后台中重新建设，提供更简洁、更直观的物模型管理能力。

### 1.2 目标
- 平台管理员可按设备**类目**（IPC摄像机、智能门铃等）管理标准物模型
- 每个类目下组织**能力模块**（工作模式、录制模式、灯光模式等），对能力进行分类
- 支持定义**属性**（枚举/数值/布尔/字符串）、**服务**（含输入输出参数）、**事件**（info/alarm/fault）
- 能力类型（属性/服务/事件）创建后可修改，适应运营灵活调整
- 统一的 CRUD 交互，编辑实时生效

## 2. 角色与职责

| 角色 | 职责 |
|------|------|
| 平台管理员 | 创建/编辑/删除设备类目 |
| 平台管理员 | 管理能力模块（新增/编辑/删除） |
| 平台管理员 | 定义标准能力（属性/服务/事件），配置数据类型和参数 |
| 厂商/开发者 | 在产品开发时引用平台定义的物模型能力 |

## 3. 名词解释

| 术语 | 说明 |
|------|------|
| **物模型** | 设备功能的数字化定义，包括属性、服务、事件，按模块分组 |
| **类目** | 设备分类（如 IPC摄像机、智能门铃），每个类目有独立的物模型体系 |
| **能力模块** | 对能力进行逻辑分组的容器（如"工作模式模块""录制模式模块"），包含模块名称和模块标识 |
| **属性（Property）** | 设备的状态参数，可读取和设置，如工作模式、录制清晰度 |
| **服务（Service）** | 设备可执行的指令，可被调用，如重启设备、格式化存储 |
| **事件（Event）** | 设备主动上报的消息，可被订阅，如移动侦测告警、电量不足 |
| **枚举型** | 数据类型，表示有限的离散值集合，如工作模式=低功耗/AOV/长电 |
| **数值型** | 数据类型，表示有范围的连续数值，如抓拍间隔 100~10000ms |
| **布尔型** | 数据类型，表示 true/false 二值状态，如事件录制开关 |
| **字符串型** | 数据类型，表示文本内容，如设备名称 |
| **标识符** | 英文命名的唯一标识，格式 `[a-zA-Z][a-zA-Z0-9_]*`，如 `work_mode` |

## 4. 核心业务流程

```
创建类目（IPC摄像机）
  → 在类目下创建能力模块（工作模式模块、录制模式模块…）
    → 在模块下添加标准能力
      → 选择能力类型（属性/服务/事件）
        → 配置数据类型和参数
          → 保存能力 → 立即生效
```

## 5. 功能架构

### 5.1 页面结构

**Page 1 — 类目列表（主页面）**
- 表格展示所有类目：类目名称、标识符、能力模块数、标准能力数、描述
- 添加/编辑/删除类目
- 点击类目进入能力管理

**Page 2 — 能力管理（类目详情）**
- 左栏（260px）：能力模块列表 + 新增/编辑/删除模块
- 右栏：能力表格，按模块 + 能力类型筛选
- 添加/编辑标准能力（3步抽屉）

### 5.2 能力类型枚举

| 类型 | 编码 | 说明 |
|------|------|------|
| 属性 | 1 | 设备状态参数，可读写/只读 |
| 服务 | 2 | 设备可执行指令，含输入/输出参数 |
| 事件 | 3 | 设备主动上报，含事件类型 |

### 5.3 数据类型枚举（属性专用）

| 类型 | key | 配置项 |
|------|------|------|
| 枚举型 | `enum` | 枚举值列表（Name + Value）、默认值 |
| 数值型 | `int` | 最小值、最大值、步长、单位、默认值 |
| 布尔型 | `boolean` | true标签、false标签、默认值 |
| 字符串型 | `string` | 最大长度（字节）、默认值 |

### 5.4 读写模式

| 模式 | 编码 | 说明 |
|------|------|------|
| 读写 | `rw` | 允许读取和设置 |
| 只读 | `ro` | 仅允许读取，不可设置 |

## 6. 详细场景分析

### 6.1 正常场景

#### S01: 创建类目
1. 管理员点击"添加类目"
2. 输入类目名称（如"IPC摄像机"）和标识符（如 `ipc_camera`）
3. 输入描述
4. 保存 → 类目列表刷新

#### S02: 创建能力模块
1. 管理员进入 IPC摄像机 类目
2. 左侧模块列表点击"+"
3. 输入模块名称（如"工作模式模块"）和模块标识（如 `work_mode_module`）
4. 保存 → 模块列表刷新

#### S03: 添加属性（枚举型）
1. 选中"工作模式模块"
2. 点击"添加标准能力"
3. Step1：选择能力类型=属性，数据类型=枚举型，填写名称、标识符、描述
4. Step2：配置枚举值（低功耗模式=0、AOV模式=1、长电模式=2、自定义=3），设默认值
5. Step3：确认预览 → 提交

#### S04: 添加属性（数值型）
1. Step1：能力类型=属性，数据类型=数值型，名称=抓拍间隔，标识符=`capture_interval`
2. Step2：最小值 100、最大值 10000、步长 100、单位 ms、默认值 500
3. 提交

#### S05: 添加服务
1. Step1：能力类型=服务，名称=重启设备，标识符=`restart`
2. Step2：配置输出参数（参数名=重启结果，标识符=`result`，类型=布尔型）
3. 提交

#### S06: 添加事件
1. Step1：能力类型=事件，名称=移动侦测，标识符=`motion_detect`
2. Step2：选择事件类型=告警(alarm)
3. 提交

#### S07: 修改能力类型
1. 编辑某个已创建的能力
2. 在 Step1 中切换能力类型（如属性→服务）
3. Step2 联动更新为服务的数据配置项
4. 提交更新

### 6.2 异常场景

| 异常 | 处理方式 |
|------|----------|
| 标识符重复 | 前端校验 + 提示"标识符已存在，请更换" |
| 标识符格式不合法 | 实时校验正则 `[a-zA-Z][a-zA-Z0-9_]*`，非法字符禁止输入 |
| 枚举值为空 | Step3 提交时校验，至少一个枚举值 |
| 数值范围不合法 | 最大值必须 > 最小值，否则提示 |
| 删除非空模块 | 二次确认弹窗，提示模块下有 N 个能力将被同步删除 |

## 7. 数据模型

### 7.1 类目 (Category)
```javascript
{
  id: Number,           // 类目ID
  name: String,         // 类目名称（1-40字符）
  identifier: String,   // 类目标识符 [a-zA-Z][a-zA-Z0-9_]*
  descr: String,        // 描述（1-200字符）
  modules: [Module]     // 能力模块列表
}
```

### 7.2 能力模块 (Module)
```javascript
{
  id: Number,           // 模块ID
  categoryId: Number,   // 所属类目ID
  name: String,         // 模块名称
  identifier: String,   // 模块标识 [a-zA-Z][a-zA-Z0-9_]*
  capabilities: [Capability]  // 能力列表
}
```

### 7.3 标准能力 (Capability)
```javascript
{
  id: Number,               // 能力ID
  moduleId: Number,         // 所属模块ID
  categoryId: Number,       // 所属类目ID
  capabilityType: Number,   // 1=属性, 2=服务, 3=事件
  capabilityName: String,   // 能力名称
  identifier: String,       // 能力标识符 [a-zA-Z][a-zA-Z0-9_]*
  descr: String,            // 描述
  capabilityJson: Object    // 能力数据定义（按类型不同）
}
```

### 7.4 capabilityJson 结构

**属性-枚举型**：
```javascript
{
  DataType: "enum", Name: "工作模式", Identifier: "work_mode",
  AccessMode: "rw",
  Enum: [{ Name: "低功耗", Value: 0 }, { Name: "AOV", Value: 1 }],
  DefaultValue: "0"
}
```

**属性-数值型**：
```javascript
{
  DataType: "int", Name: "抓拍间隔", Identifier: "capture_interval",
  AccessMode: "rw", Min: 100, Max: 10000, Step: 100, Unit: "ms", DefaultValue: "500"
}
```

**属性-布尔型**：
```javascript
{
  DataType: "boolean", Name: "事件录制", Identifier: "event_record",
  AccessMode: "rw",
  Enum: [{ Name: "开启", Value: 1 }, { Name: "关闭", Value: 0 }],
  DefaultValue: "1"
}
```

**属性-字符串型**：
```javascript
{
  DataType: "string", Name: "设备名称", Identifier: "device_name",
  AccessMode: "rw", limit: 64, DefaultValue: ""
}
```

**服务**：
```javascript
{
  DataType: "service", Name: "重启设备", Identifier: "restart",
  InputParam: {},      // 输入参数（可为空对象）
  OutputParam: { Name: "重启结果", Identifier: "result", DataType: "boolean" }
}
```

**事件**：
```javascript
{
  DataType: "event", Name: "移动侦测", Identifier: "motion_detect",
  EventType: "alarm",  // info | alarm | fault
  OutputParam: {}      // 输出参数（可为空对象）
}
```

## 8. 业务规则

| 编号 | 规则 |
|------|------|
| R01 | 类目名称不能为空，最长 40 字符 |
| R02 | 标识符全局唯一（类目+模块+能力），格式 `[a-zA-Z][a-zA-Z0-9_]*` |
| R03 | 能力类型创建后可修改，修改后 Step2 数据定义重置 |
| R04 | 模块删除前校验其下能力数，二次确认 |
| R05 | 枚举型至少需要 1 个枚举值 |
| R06 | 数值型最大值必须 > 最小值 |
| R07 | 删除类目前校验其下是否有模块和能力，二次确认 |
| R08 | 能力类型为属性时必须选择数据类型 |
| R09 | 属性默认值必须符合数据类型规则 |
| R10 | 能力类型为服务时，输入参数和输出参数为可选 |
| R11 | 能力类型为事件时，事件类型必选 |
| R12 | 标识符不可包含中文、空格、特殊字符 |

```
  └── Mock 数据层（reactive 单例，内存存储）

  └── 作为 iot-platform 的参考实现

```
