## 一、业务背景

产品上线后，运营经常需要对比不同版本的效果来做出优化决策——不仅是页面布局和文案，还包括套餐定价、支付流程、权益组合等。运营需要通过 A/B Test 对这些变量进行数据驱动的对比验证。

**典型应用场景**：页面版本对比、套餐定价对比、运营策略对比。

**核心目标**：通过数据驱动验证不同版本/定价/策略对用户转化率的影响，找出最优方案并全量应用。

---

## 二、角色与职责

| 角色 | 职责 | 关键约束 |
|------|------|----------|
| APP（客户端） | 内置实验 Key + 变体内容，根据后台分流结果渲染对应版本 | 不参与流量分配决策，只管渲染和埋点上报 |
| 后台（iot-platform） | 创建实验、配置流量比例、启动/暂停/结束、数据统计、声明胜出 | 只管 A/B 分流的开关和比例，不管 A/B 具体内容 |
| 运营 | 配置实验参数、观察数据、决策胜出变体 | 不接触代码 |
| 用户 | 无感知参与实验，被 Hash 分到 A 或 B | 同一用户始终看到同一变体 |

---

## 三、名词解释

| 术语 | 说明 |
|------|------|
| 实验 | 一次 A/B Test，包含变体A和变体B，面向全部用户运行 |
| 变体 | 实验中的一个版本，APP端原生实现变体内容，后台记录变体标识 |
| 分桶 | 通过 Hash(user_id + experiment_key) 将用户实时分到 A 或 B 变体 |
| 分流 | 用户请求时，后台计算其所属变体并返回，APP据此渲染对应版本 |
| 转化率 | 支付成功数 / 页面曝光数 |
| 置信度 | 统计显著性的衡量标准，≥95% 可下结论 |

## 四、业务实体说明

### 核心实体

**实验（Experiment）**
- 一次 A/B Test 的配置记录
- 关键属性：名称、唯一Key、变体A/B标识、流量比例、状态
- 生命周期：待启动 → 运行中 ⇄ 已暂停 → 已结束

**变体（Variant）**
- A（对照组）：原始版本
- B（实验组）：新版本
- 变体具体内容由 APP 端原生实现，后台仅记录 A/B 标识

**埋点事件（Event）**
- 用户行为记录：页面曝光、套餐选择、支付点击、支付成功
- 携带实验信息（experiment_key、variant）用于数据分析

### 实体关系

```
实验 1──2 变体（A/B）
实验 1──N 埋点事件
用户 1──N 埋点事件
```

## 五、核心业务流程

### 5.1 实验全生命周期（正常场景）

```
APP发版（内置Key+变体内容）
  → 运营创建实验（填写名称、Key、流量比例）
  → 运营启动实验
  → 用户进入落地页 → 后台Hash分流 → APP渲染对应变体 → 上报曝光
  → 用户浏览 → 选择套餐 → 支付 → 上报转化
  → 运营查看数据（可暂停/恢复）
  → 数据达标 → 声明胜出 → 全量切换
  → APP下次发版清理实验代码
```

### 5.2 用户分流流程（正常场景）

```
APP请求分流接口（携带user_id + experiment_key）
  → 后台查询是否有运行中的实验
  → 无 → 返回默认版本
  → 有 → Hash(user_id + experiment_key) % 100
    → bucket < trafficA → 返回变体A
    → bucket >= trafficA → 返回变体B
  → APP根据返回的变体标识渲染对应原生页面内容
```

## 七、详细功能分析

### 7.1 实验列表页

**功能应用场景**：运营查看所有实验的概况，快速了解实验状态和数据。

**正常场景**：
- SC-01：展示所有实验列表，包含实验名称、Key、流量比例、状态、创建时间
- SC-02：支持按状态筛选（全部/待启动/运行中/已暂停/已结束）
- SC-03：每行显示该实验的实时核心数据（曝光、转化率、收入）
- SC-04：点击「创建实验」打开创建抽屉
- SC-05：根据状态显示不同的操作按钮

**异常场景**：
- EX-01：实验列表为空 → 显示空态提示"暂无实验，点击创建第一个实验"
- EX-02：列表加载失败 → 提示"加载失败，请刷新重试"，提供刷新按钮
- EX-03：删除进行中的实验 → 二次确认弹窗"实验运行中，确定删除？删除后数据不可恢复"

### 7.2 创建/编辑实验

**功能应用场景**：运营创建新实验或修改已有实验配置。

**正常场景**：
- SC-01：创建时所有字段为空，标题「创建实验」
- SC-02：编辑时回填已有数据，标题「编辑实验」，Key 置灰不可修改
- SC-03：流量比例通过滑块调节，默认50/50，实时显示百分比数字
- SC-04：填写完整后点击保存，创建成功并关闭抽屉，列表刷新

**异常场景**：
- EX-01：必填字段为空（名称、Key）→ 表单校验提示"请填写XXX"
- EX-02：实验Key重复 → 提示"该Key已被使用，请更换"
- EX-03：Key格式非法（含特殊字符或大写）→ 提示"仅支持小写字母、数字、下划线"
- EX-04：保存时网络异常 → 提示"保存失败，请重试"，保留已填数据不关闭抽屉
- EX-05：编辑时实验已被他人删除 → 提示"该实验不存在或已被删除"

### 7.3 实验状态管理

**功能应用场景**：运营控制实验的生命周期。实验的启动、暂停、恢复、结束均由运营在后台手动操作，系统只提供数据看板和判断条件作为参考。

**正常场景**：
- SC-01：待启动 → 点击启动 → 确认后分流生效，状态变更为运行中
- SC-02：运行中 → 点击暂停 → 确认后所有用户返回默认版本，状态变更为已暂停
- SC-03：已暂停 → 点击恢复 → 继续按比例分流，状态变更为运行中
- SC-04：运行中/已暂停 → 点击结束 → 确认后状态变更为已结束
- SC-05：已结束 → 点击应用胜出 → 确认后将胜出变体设为全量默认

**异常场景**：
- EX-01：同时多个运营操作同一实验 → 以先操作为准，后操作提示"实验状态已变更，请刷新"
- EX-02：启动时无可用APP版本 → 提示"未检测到支持此实验Key的APP版本，启动后可能无效果"

### 7.4 分桶路由

**功能应用场景**：APP请求分流接口，获取当前用户应看到的变体。

**正常场景**：
- SC-01：用户有对应运行中实验 → 返回 experiment_key + variant
- SC-02：用户无对应实验 → 返回 variant=default，使用默认版本
- SC-03：同一用户同一实验 → Hash结果始终相同，体验一致

**异常场景**：
- EX-01：user_id 为空 → 返回400 + 提示"缺少必要参数"
- EX-02：experiment_key 不存在 → 返回 variant=default
- EX-03：接口超时（>3s）→ APP端使用默认版本兜底
- EX-04：高并发场景（>10000 QPS）→ Hash计算纯CPU运算，纳秒级，不会成为瓶颈

### 7.5 数据看板

**功能应用场景**：运营查看实验数据，判断变体优劣。

**正常场景**：
- SC-01：顶部显示实验基本信息（流量比例、运行天数）
- SC-02：4个指标卡片并排：曝光UV、选择率、转化率、总收入
- SC-03：A/B变体对比表格，每项指标显示差异百分比
- SC-04：转化漏斗展示 曝光→选择→支付点击→支付成功 各步数据
- SC-05：每日趋势折线图（横轴日期，纵轴转化率，A线 vs B线）

**异常场景**：
- EX-01：实验刚启动数据量少 → 结论区显示"数据不足，继续收集"，提示需≥500转化事件
- EX-02：运行天数不足7天 → 结论区提示"运行天数不足，建议继续观察"
- EX-03：置信度<95% → 提示"差异不显著，可能为随机波动"
- EX-04：流量分配不均（max/min > 1.2）→ 提示"流量分配异常，请检查分流逻辑"
- EX-05：数据延迟（>5分钟未收到新事件）→ 提示"数据可能存在延迟"

## 八、业务规则

### 8.1 实验管理规则

| 编号 | 规则 | 说明 |
|------|------|------|
| R01 | Key唯一性 | 实验Key在活跃实验（待启动/运行中/已暂停）中唯一，创建时校验；已结束实验的Key可复用，创建新实验时数据隔离 |
| R02 | APP前置 | APP必须先内置对应Key和变体内容，后台才能启动实验 |
| R03 | 编辑限制 | 运行中/已暂停的实验不可修改Key |
| R04 | 删除限制 | 运行中的实验需先暂停才能删除 |
| R05 | 流量比例 | 流量A占比范围 1-99，默认50（A:B = 50:50） |

### 8.2 分流规则

| 编号 | 规则 | 说明 |
|------|------|------|
| R06 | 实时计算 | 用户进入页面时才计算分桶，不预分配 |
| R07 | 一致性 | 同一用户+同一实验，Hash结果永远相同 |
| R08 | 新用户适配 | 新用户自动落入某个桶，无需预分配 |
| R09 | 暂停回退 | 实验暂停/结束后，所有用户返回默认版本 |
| R10 | Hash算法 | hash = crc32(user_id + experiment_key) % 100 |

### 8.3 数据判断规则

| 编号 | 规则 | 说明 |
|------|------|------|
| R11 | 转化事件 | 每组≥500个转化事件可下结论 |
| R12 | 运行天数 | ≥7天可下结论 |
| R13 | 置信度 | ≥95%可下结论 |
| R14 | 胜出标准 | 三条同时满足 + 转化率差异显著 → 可声明胜出 |

## 九、异常场景总览

| 分类 | 场景 | 处理方式 |
|------|------|------|
| 网络异常 | 分流接口超时 | APP使用默认版本兜底 |
| 网络异常 | 看板数据加载失败 | 保留上次缓存数据 + 提示刷新 |
| 网络异常 | 创建/编辑保存失败 | 提示重试，保留表单数据 |
| 数据异常 | 实验Key不存在 | 返回默认版本，不报错 |
| 数据异常 | user_id为空 | 返回400错误 |
| 数据异常 | 流量分配严重不均 | 看板提示告警 |
| 数据异常 | 埋点参数缺失 | 丢弃该条 + 后台告警 |
| 数据异常 | 重复事件（1min内） | 保留第一条，丢弃后续 |
| 并发异常 | 多人同时操作同一实验 | 先到先得，后操作提示刷新 |
| 极限场景 | 实验列表超100条 | 分页加载，每页20条 |
| 极限场景 | 单实验数据事件超百万 | 数据聚合预计算，不实时查明细 |

## 十、跨角色联动分析

### APP ↔ 后台协作

| 环节 | APP职责 | 后台职责 |
|------|------|------|
| 实验准备 | 内置Key + 变体原生页面内容 | 提供实验管理后台 |
| 用户分流 | 进入页面时调用分流接口 | Hash计算并返回变体标识 |
| 页面渲染 | 根据返回的变体渲染对应原生页面 | — |
| 埋点上报 | 上报曝光/选择/支付事件 | 存储事件数据，提供看板 |
| 实验结束 | 下次发版清理实验代码 | 声明胜出，全量切换 |
| 兜底处理 | 接口异常时使用默认版本 | — |

### 关键约束
- **APP不发版，后台无法启动新实验**（APP需先内置Key和变体内容）
- **分桶由后台计算，APP不参与分配决策**
- **变体内容由APP原生控制，后台不干预具体UI**
