## 一、业务背景

产品上线后，运营经常需要对比不同版本的效果来做出优化决策——不仅是页面布局和文案，还包括套餐定价、支付流程、权益组合等。运营需要通过 A/B Test 对这些变量进行数据驱动的对比验证，而非凭经验或直觉判断。

**A/B Test 的典型应用场景**：
- 页面版本对比：不同落地页布局、文案、配图的转化效果
- 套餐定价对比：不同价格方案（如 ¥9.9 vs ¥19.9）的用户选择和转化
- 运营策略对比：不同优惠力度、不同权益组合的转化效果

**要解决的痛点**：
- 产品经理和运营无法量化不同设计/定价/策略对转化的影响
- 改版和调价决策依赖主观判断，缺乏数据支撑
- 没有系统化的实验管理手段，实验散落在代码中难以管理和追溯

**产品目标**：
- 提供通用的 A/B Test 实验管理平台，支持创建、启停、数据分析和胜出应用
- 通过 Hash 实时分流保证实验的数学严谨性和零存储成本
- 数据看板直观展示 A/B 对比，辅助运营做出统计显著的决策

## 二、名词解释

| 术语 | 说明 |
|------|------|
| 实验 | 一次 A/B Test，包含变体A和变体B，面向全部用户运行 |
| 变体 | 实验中的一个版本（A=对照组，B=实验组），APP端原生实现具体内容 |
| 分桶 | 通过 Hash(user_id + experiment_key) % 100 将用户实时分到 0~99 的桶 |
| 分流 | 用户请求时，后台计算其所属变体并返回，APP 据此渲染对应版本 |
| 转化率 | 支付成功数 / 页面曝光数，核心衡量指标 |
| 置信度 | 统计显著性的衡量标准，≥95% 可下结论 |

## 三、业务实体说明

### 3.1 核心实体

**实验（Experiment）**
- 属性：名称、唯一Key、流量比例、状态（待启动/运行中/已暂停/已结束）、胜出变体、时间戳
- Key由运营与APP开发者约定，创建后不可修改
- 生命周期：待启动 → 运行中 ⇄ 已暂停 → 已结束

**变体（Variant）**
- A（对照组）：通常为线上当前版本
- B（实验组）：新版本/待测试版本
- 变体内容由APP端原生实现，后台不管理具体内容

**埋点事件（Event）**
- 记录用户行为：页面曝光、套餐选择、支付点击、支付成功
- 携带实验上下文（experiment_key、variant）

### 3.2 实体关系

```
实验 1──2 变体（A/B）
实验 1──N 实验事件（数据累计）
用户 N──M 实验（一个用户可能参与多个实验）
```

## 五、业务规则

### 5.1 实验管理

| 编号 | 规则 | 说明 |
|------|------|------|
| R01 | Key唯一性 | 实验Key在活跃实验（待启动/运行中/已暂停）中唯一，创建时校验；已结束实验的Key可复用，复用Key创建实验与原实验数据隔离 |
| R02 | APP前置 | APP必须先内置对应Key和变体内容，后台才能启动实验 |
| R03 | 编辑限制 | 运行中/已暂停的实验不可修改Key |
| R04 | 删除限制 | 运行中的实验需先暂停才能删除 |
| R05 | 流量范围 | 流量A占比 1~99%，默认50%（A:B=50:50） |

### 5.2 用户分流

| 编号 | 规则 | 说明 |
|------|------|------|
| R06 | 实时计算 | 用户进入页面时实时计算分桶，不预分配 |
| R07 | 一致性 | 同一用户+同一实验+不变比例 → 始终看到同一变体 |
| R08 | 新用户适配 | 实验运行期间的新用户自动通过Hash落入桶中 |
| R09 | 暂停回退 | 暂停/结束后所有用户返回默认版本（变体A） |
| R10 | 仅统计访问者 | 未进入落地页的用户不触发分流，不计入实验分母 |

### 5.3 数据判断

| 编号 | 规则 | 说明 |
|------|------|------|
| R11 | 转化事件量 | 每组≥500个转化事件 |
| R12 | 运行天数 | ≥7天 |
| R13 | 置信度 | ≥95% |
| R14 | 胜出条件 | R11+R12+R13同时满足 + A/B转化率存在显著差异 |

## 七、详细功能描述

### 7.1 实验管理

#### 功能应用场景
运营在后台创建和管理 A/B Test 实验，控制实验生命周期。

#### 实验列表
- 展示所有实验，支持按状态筛选（全部/待启动/运行中/已暂停/已结束）
- 每行显示：实验名称、Key、流量比例、状态标签、创建时间
- 每行尾部嵌入该实验实时核心数据（曝光数、转化率、收入）
- 根据实验状态显示对应操作按钮组合

#### 创建/编辑实验
- 通过右侧滑出抽屉表单完成，宽度 560px
- 编辑时实验Key置灰不可修改
- 流量分配使用滑块调节，实时显示A/B百分比
- 必填校验：名称、Key为空时提示
- 变体A/B的具体内容由APP端原生实现，不在后台配置

#### 实验操作

实验全生命周期由**运营手动控制**。系统提供数据看板和结论判断条件作为参考，但不自动启停实验。运营根据数据判断后手动执行操作。

| 操作 | 适用状态 | 行为 |
|------|------|------|
| 启动 | 待启动 | 确认后分流开始生效 |
| 暂停 | 运行中 | 确认后所有用户回到默认版本 |
| 恢复 | 已暂停 | 直接恢复，继续按比例分流 |
| 结束 | 运行中/已暂停 | 确认后选择胜出变体，实验归档 |
| 应用胜出 | 已结束有胜出 | 将胜出变体设为全量默认 |
| 删除 | 所有状态 | 二次确认，运行中先暂停再删 |

---

### 7.2 分流路由

#### 功能应用场景
用户在APP中进入实验覆盖的页面时，后台实时计算该用户应看到哪个版本。

#### 功能描述
- APP进入实验页面时，调用后台分流接口
- 后台根据 user_id 和 experiment_key 做 Hash 运算，返回变体标识
- 同一用户+同一实验保证结果一致（Hash确定性）
- 无实验或实验未运行时返回默认版本
- 接口超时或异常时，APP使用默认版本兜底

#### 分桶原理
Hash 将 (user_id + experiment_key) 映射到 0~99 共100个桶。如流量A=50%，则桶0~49为A，桶50~99为B。调整流量只需修改分界值，无需重新分配。

---

### 7.3 数据看板

#### 功能应用场景
运营查看实验运行数据，判断变体A和变体B哪个效果更好。

#### 页面组成

| 区域 | 内容 |
|------|------|
| 信息栏 | 实验名称 + 状态标签 + 流量比例 + 已运行天数 |
| 指标卡片 | 总曝光UV、总转化UV、整体转化率、总收入（4列） |
| 对比区 | A/B各指标并排对比，差异百分比（绿色↑/红色↓） |
| 转化漏斗 | 曝光 → 选择套餐 → 支付点击 → 支付成功，A/B两列对比 |
| 趋势图 | 每日转化率折线图（横轴日期、纵轴转化率、A线 vs B线） |
| 结论区 | 三个判断条件的达标状态 + 最终结论文字 |

#### 结论展示

| 数据状态 | 结论显示 |
|------|------|
| 三项条件未全达标 | "数据不足，继续收集" + 各条件当前进度 |
| 三项达标，差异不显著 | "无显著差异，任一变体均可" |
| 三项达标，B显著优于A | 绿色横幅"★ 变体B胜出，转化率+XX%，建议应用" |
| 三项达标，A显著优于B | 绿色横幅"★ 变体A胜出，转化率+XX%，建议应用" |

### 7.4 埋点事件

#### 功能应用场景
APP端在用户完成关键行为时上报事件数据，供数据看板统计分析。

#### 事件定义

每次实验按具体业务定义事件。APP端在用户完成关键行为时上报事件数据，所有事件必须携带公共参数（13项）和实验上下文参数（experiment_key、variant）。具体事件清单和业务参数参见各实验案例。

## 九、异常说明

| 分类 | 场景 | 处理方式 |
|------|------|------|
| 网络异常 | 分流接口超时 | APP使用默认版本兜底 |

| 边界条件 | 运行中修改流量比例 | 允许但可能影响部分用户一致性，建议慎用 |

## 十、实验案例：全能AI落地页 A/B Test

> 以下为真实实验案例，展示如何用本系统配置一次完整的 A/B Test。案例与上方功能描述独立，便于研发按功能+案例对照查阅。

### 10.1 案例背景

全能AI落地页是用户开通智能看护服务的核心转化节点。本期对落地页进行 A/B Test，对比当前版本（A）与新版（B）的转化效果。

| 维度 | 说明 |
|------|------|
| 实验页面 | 全能AI服务介绍开通页（落地页） |
| 覆盖用户 | 未开通全能AI的用户（已开通用户直接进入AI配置页，不走落地页） |
| 实验周期 | ≥ 7天，每组转化事件 ≥ 500 |

### 10.2 实验配置

| 属性 | 值 |
|------|------|
| 实验Key | `ai_landing_ab` |
| 实验名称 | 全能AI落地页A/B Test |
| A（对照组） | 当前线上落地页版本 |
| B（实验组） | 新版落地页版本 |

> 变体A/B的具体页面内容由APP/H5实现，运营后台仅管理实验配置和流量分配。

### 10.3 分流规则

**分流算法**：`Hash(user_id + "ai_landing_ab") % 100` → 桶 0~99

**海内外独立分流**：

| 用户区域 | 判定条件 | A组桶区间 | B组桶区间 | A:B |
|------|------|------|------|------|
| 国内（CN） | country = "CN" | 0 ~ 49 | 50 ~ 99 | 50:50 |
| 海外（非CN） | country ≠ "CN" | 0 ~ 49 | 50 ~ 99 | 50:50 |

> 海内外各自独立分流，同一用户+同一实验始终看到同一变体。暂停后全部回A，异常兜底返回A。

### 10.4 埋点事件

#### 事件清单

| 事件ID | 事件名称 | 触发时机 | event_type | 优先级 |
|------|------|------|------|------|
| ab_landing_entry | 入口点击 | 用户从任意入口进入落地页，首屏可见 | 1（额外参数） | P0 |
| ab_landing_close | 点击关闭 | 用户点击关闭/返回按钮离开落地页 | 1（额外参数） | P0 |
| ab_activate_click | 点击开通 | 用户勾选协议后点击「立即开通」按钮 | 1（额外参数） | P0 |
| ab_pay_success | 支付成功 | 收到支付成功回调 | 1（额外参数） | P0 |
| ab_pay_fail | 支付失败 | 支付失败或用户取消支付 | 1（额外参数） | P0 |

### 10.6 核心指标

| 序号 | 指标 | 计算方式 | 周期 |
|------|------|------|------|
| 1 | A/B 端到端转化率 | ab_pay_success UV / ab_landing_entry UV（按variant） | 日 |

### 10.7 胜出判定

同第五章业务规则（R11~R14）：每组转化 ≥ 500 + 运行 ≥ 7天 + 置信度 ≥ 95%，三项达标后对比转化率声明胜出。

```
APP发版（内置Key+两版落地页） → 运营创建实验 → 启动 → 运行 ≥7天 → 数据达标 → 声明胜出 → 全量应用 → APP下次发版收敛
```
