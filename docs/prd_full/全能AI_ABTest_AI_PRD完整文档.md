## 一、业务背景与目标

全能AI落地页已上线，包含两个套餐（¥9.9 全能AI / ¥19.9 全能AI Pro）。当前所有未开通用户看到的是同一个页面。运营需要通过 A/B Test 对比不同落地页版本的转化效果，用数据驱动页面优化决策。

**核心目标**：验证不同落地页版本对用户转化率的影响，找出最优版本并全量应用。

---

## 二、角色与职责

| 角色 | 职责 | 关键约束 |
|------|------|----------|
| **APP (carecam_pro)** | 内置实验 Key + 变体A/B页面内容，根据后台分流结果渲染对应版本 | 不参与流量分配决策，只管渲染和埋点上报 |
| **后台 (iot-platform)** | 创建实验、配置流量比例、启动/暂停/结束、数据统计、声明胜出 | 只管 A/B 分流的开关和比例，不管 A/B 具体内容 |
| **运营** | 配置实验参数、观察数据、决策胜出变体 | 不接触代码 |
| **用户** | 无感知参与实验，被 Hash 分到 A 或 B | 同一用户始终看到同一变体 |

| 序号 | 模块 | 功能 | 负责人 |

## 四、实验全生命周期

```
 APP 侧：  [发版内置Key+内容] ───────────────────────────────▶ [下次发版清理]
             │                                                  │
 后台侧：    │   [创建] → [启动] → [数据观察] → [声明胜出] → [结束]  │
             │      │        │           │          │          │
 用户侧：    └──────┴────────┴───────────┴──────────┴──────────┘
                       │           │
                    分流生效    看到A/B版本
```

### 各阶段说明

| 阶段 | 谁 | 做什么 | 实验状态 |
|------|-----|------|----------|
| 0. 发版 | APP开发 | 内置 Key+变体内容，发版 | — |
| 1. 创建 | 运营 | 填写名称、Key、流量比例 | 待启动 |
| 2. 启动 | 运营 | 点击启动，分流开始生效 | 运行中 |
| 3. 分流 | 系统 | 用户进入页面 → Hash分流 → APP渲染A/B → 上报曝光 | 运行中 |
| 4. 转化 | 系统 | 用户选择套餐 → 支付 → 上报转化 | 运行中 |
| 5. 观察 | 运营 | 查看数据，可随时暂停/恢复 | 运行中/已暂停 |
| 6. 结束 | 运营 | 对比数据，声明胜出，全量切换 | 已结束 |
| 7. 收敛 | APP开发 | 下次发版移除实验代码，胜出变体设为默认 | 已结束 |

**关键原则**：APP 不发版，后台无法启动新实验。因为 APP 需要先内置实验的 Key 和变体内容。

## 五、分流机制（业务说明）

### 核心概念

- **实时计算**：用户进入落地页时才计算他属于 A 还是 B，不提前分配
- **数学保证均匀**：通过 Hash(用户ID + 实验Key) % 100 将用户均匀分到 100 个桶中，前 N 个桶为 A，后 100-N 个为 B
- **同一用户始终一致**：同一用户 + 同一实验 → Hash 结果永远相同 → 看到的版本不变
- **新用户自动适配**：无需预分配，新用户自然落入某个桶

### 只统计「访问者」

- **只有进入落地页的用户才触发分流**，才计入实验分母
- 从未访问落地页的用户不参与实验，不影响结论
- 未被分配 ≠ 数据缺失，而是设计意图：实验只关心「看到页面」的用户

### 状态与分流行为

| 状态 | 分流行为 | 用户看到 |
|------|----------|----------|
| 待启动 | 不参与分流 | 默认版本（A） |
| 运行中 | 按比例分流 A/B | A 或 B |
| 已暂停 | 全部返回 A | 默认版本（A） |
| 已结束 | 胜出变体已定 → 全部返回胜出；否则全部返回 A | 胜出变体 |

## 六、实验管理功能（后台）

### 6.1 实验列表页

**页面路径**：套餐管理 / A/B Test（菜单结构见项目 Layout 配置）

**列表字段**：

| 字段 | 说明 |
|------|------|
| 实验名称 | 运营可读名称，如「全能AI落地页v1 vs v2」 |
| 实验Key | 系统唯一标识，与 APP 内置 Key 一致，如 `ai_landing_test_01` |
| 流量分配 | A x% / B y% |
| 状态 | 待启动 / 运行中 / 已暂停 / 已结束 |
| 创建时间 | 创建日期 |
| 操作 | 启动、编辑、暂停/恢复、查看数据、应用胜出、删除 |

**操作说明**：

| 操作 | 适用状态 | 行为 |
|------|------|------|
| 启动 | 待启动 | 确认后分流开始生效 |
| 编辑 | 待启动/运行中/已暂停 | 可修改名称、流量比例等 |
| 暂停 | 运行中 | 所有用户回到默认版本 |
| 恢复 | 已暂停 | 继续按比例分流 |
| 数据 | 全部 | 跳转数据看板页 |
| 应用胜出 | 已结束且有胜出 | 将胜出变体设为全量默认 |
| 删除 | 全部 | 确认后不可撤销 |

### 6.2 创建/编辑实验

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 实验名称 | 文本 | 是 | 如「全能AI落地页v1 vs v2」 |
| 实验Key | 文本 | 是 | 唯一标识，仅小写字母/数字/下划线，创建后不可修改 |
| 流量A占比 | 滑块(0-100) | — | 默认50，即 A:B = 50:50 |
| 备注 | 多行文本 | 否 | 实验目的和假设说明 |

> 注：变体 A/B 的具体内容（页面 UI、文案、布局等）由 APP 端写死，不在后台配置。

## 七、数据看板（后台）

### 7.1 看板概览

**页面路径**：A/B Test → 点击实验「数据」→ 数据看板

**页面组成**：

| 区域 | 内容 |
|------|------|
| 状态栏 | 实验状态标签 + 分群 + 流量比例 + 运行天数 |
| 指标卡片 | 4 个对比卡片：曝光UV、选择率、转化率、总收入 |
| 转化漏斗 | A/B 两列完整漏斗（曝光 → 选择 → 支付点击 → 支付成功） |
| 柱状图 | 漏斗各步骤 A/B 对比柱状图 |
| 胜出横幅 | 实验结束时绿色横幅，显示胜出变体 + 提升幅度 |

### 7.2 核心指标

**实验概览指标**：

| 指标 | 计算方式 |
|------|------|
| 总曝光 UV | 进入实验页面的独立用户数 |
| 总转化 UV | 支付成功的独立用户数 |
| 整体转化率 | 总转化UV / 总曝光UV |
| 总收入 | 支付金额汇总 |

**变体对比指标**：

| 指标 | 说明 |
|------|------|
| 页面曝光(UV) | 各变体去重用户数 |
| 套餐选择率 | 选择套餐数 / 曝光数 |
| 支付转化率 | 支付成功数 / 曝光数 |
| 总收入 | 变体带来的总收入 |

### 7.3 数据判断标准

| 条件 | 标准 |
|------|------|
| 每组转化事件 | ≥ 500 可下结论 |
| 运行天数 | ≥ 7 天 |
| 置信度 | ≥ 95% |

## 八、埋点事件定义

### 8.1 事件列表

| 事件名 | 触发时机 | 优先级 |
|------|------|------|
| `experiment_view` | 落地页加载完成，首屏可见 | P0 |
| `plan_select` | 用户点击选择套餐 | P0 |
| `pay_click` | 用户点击「立即开通」或支付按钮 | P0 |
| `pay_success` | 收到支付成功回调 | P0 |

### 8.2 公共参数（所有事件必带）

| 参数 | 说明 |
|------|------|
| user_id | 用户唯一标识 |
| experiment_key | 实验标识 |
| variant | 变体 A/B |
| segment | 用户分群 |

### 8.3 事件特有参数

| 事件 | 特有参数 |
|------|------|
| experiment_view | 页面URL |
| plan_select | 套餐类型(basic/pro)、套餐价格 |
| pay_click | 套餐类型、套餐价格 |
| pay_success | 套餐类型、套餐价格、订单ID |
