# 工作流程指南

## 目录结构说明

### 1. design_standard/ - 标准文件库
保存 UI 设计标准和 PRD 设计标准
- `ui-design_app.md` - App UI 设计标准
- `prd_design_app.md` - App PRD 设计标准
- `ui-design_web.md` - Web UI 设计标准

### 2. prd_dec/ - 简要需求描述
用户提供的简洁需求说明，包含核心功能描述
- 例如：`carecampro_prd.md`、`Platform_prd.md`
- 这些文件包含用户的初步想法和需求概述

### 3. prd_full/ - 完整需求文档
根据简要需求 + 设计标准生成的完整 PRD 文档
- 经过用户确认后的最终版本
- 例如：`AOV工作模式PRD.md`
- 这些文档结构完整、格式规范、内容详细

## 工作流程

1. **接收简要需求**
   - 用户在 `prd_dec/` 中提供简要需求

2. **生成页面设计**
   - 根据简要需求 + `design_standard/` 中的 UI 标准生成 `.html` 设计稿
   - 保存到 `design/` 目录
   - **等待用户确认页面设计**

3. **用户确认页面后，生成完整 PRD**
   - 根据已确认的页面设计 + `design_standard/` 中的 PRD 标准生成完整文档
   - 保存到 `prd_full/` 目录

4. **用户最终确认 PRD**

## 重要提示

- **每次新会话都要遵循这个流程**
- **不要混淆三个目录的用途**
- **prd_full 中的文档应该是完整、规范、经过确认的版本**
- **参考 design_standard 中的标准来保持一致性**

## Web 端通用交互规则

### 列表页弹窗规则
- 所有 web 端系统功能中，列表的**添加弹窗**和**编辑弹窗**内容完全一致
- 唯一区别是弹窗标题文案不同：
  - 添加：**添加 XX**
  - 编辑：**编辑 XX**
- 写 PRD 或做设计时，只需描述一次弹窗内容，自动适用于添加和编辑两种场景
- 举一反三：凡是列表类功能，无需重复说明添加/编辑弹窗的字段，默认两者一致
