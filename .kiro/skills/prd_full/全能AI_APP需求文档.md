## 一、业务背景

全能AI 是面向未开通云服务用户的 AI 功能落地页，展示两个套餐（¥9.9 全能AI / ¥19.9 全能AI Pro）。为了提升转化率，运营需要通过 A/B Test 对比不同落地页版本的效果，用数据驱动页面优化。

本需求文档描述 APP 端（carecam_pro）在「全能AI」模块中需要实现的功能，包括落地页展示、A/B 实验参与、套餐购买、以及埋点上报。

> APP 与后台的完整协作流程详见 [`.kiro/specs/abtest-flow.md`](../specs/abtest-flow.md)。

---

## 二、角色与职责

| 角色 | 职责 |
|------|------|
| **APP** | 内置实验 Key + 变体A/B页面内容；请求后台分流接口获取当前用户归属变体；渲染对应版本；埋点上报 |
| **后台** | 创建/管理实验、配置流量比例、提供分流结果、接收埋点数据、数据统计看板 |
| **运营** | 在后台配置实验、观察数据、决策胜出变体 |

**核心原则**：APP 只管「当前用户看 A 还是 B」以及「渲染什么内容」，不参与流量分配的决策。

## 三、功能架构

```
全能AI 落地页
├── 1. 入口
│   └── 从设备卡片「云服务」/ 服务商城「全能AI」进入
├── 2. A/B 分流
│   ├── 请求后台获取当前用户变体（A/B）
│   └── 根据变体渲染对应页面内容
├── 3. 页面内容（两个变体）
│   ├── 变体A：默认版落地页
│   └── 变体B：实验版落地页
├── 4. 套餐展示与选择
│   ├── ¥9.9 全能AI 套餐
│   └── ¥19.9 全能AI Pro 套餐
├── 5. 支付购买
│   ├── 内购流程（Google Play / Apple Store）
│   └── 支付成功 → 开通服务
└── 6. 埋点上报
    ├── 页面曝光
    ├── 套餐选择
    ├── 支付点击
    └── 支付成功
```

## 四、入口

### 4.1 入口场景

| 入口 | 触发条件 | 说明 |
|------|------|------|
| 设备卡片-云服务 | 用户在首页设备卡片点击云服务入口，且当前设备未开通云服务 | 从设备维度引导开通 |
| 服务商城-全能AI | 用户在「我的」→「服务商城」→ 选择「全能AI」分类 | 从商城维度浏览购买 |

### 4.2 进入落地页流程

```
用户点击入口
    ↓
APP 请求后台分流接口（传入 user_id + experiment_key）
    ↓
后台返回：variant = "A" 或 "B"
    ↓
APP 根据 variant 渲染对应页面内容
    ↓
页面加载完成 → 上报 experiment_view 事件
```

## 五、A/B 实验机制

### 5.1 APP 需要做的事

1. **内置实验 Key**：如 `ai_landing_test_01`，与运营在后台配置的 Key 一致
2. **内置变体内容**：变体 A 和变体 B 的完整页面 UI（布局、文案、配色、交互等）
3. **请求分流**：进入落地页时，调后台接口获取当前用户属于 A 还是 B
4. **渲染对应版本**：根据分流结果显示变体 A 或变体 B
5. **上报埋点**：曝光、选择、支付等关键行为上报

### 5.2 实验 Key 定义

| 实验 Key | 说明 | 变体A | 变体B |
|------|------|------|------|
| `ai_landing_test_01` | 全能AI落地页 v1 vs v2 | 当前默认版落地页 | 新版落地页（由产品定义差异） |

### 5.3 变体差异（产品需定义）

> 以下为示例，具体差异由产品和设计确定。

| 对比维度 | 变体A（默认版） | 变体B（实验版） |
|------|------|------|
| 页面布局 | 现有布局 | 新布局（若有） |
| 头部文案 | 现有文案 | 新文案（若有） |
| 套餐卡片样式 | 现有样式 | 新样式（若有） |
| 价格展示 | 现有展示 | 新展示（若有） |
| CTA按钮 | 现有按钮 | 新按钮（若有） |

### 5.4 分流与状态

| 后台实验状态 | APP 行为 |
|------|------|
| 待启动 / 无实验 | 渲染默认版本（变体A） |
| 运行中 | 按分流结果渲染 A 或 B |
| 已暂停 | 全部渲染默认版本（变体A） |
| 已结束（有胜出） | 全部渲染胜出变体 |
| 已结束（无胜出） | 全部渲染默认版本（变体A） |

## 七、购买支付流程

```
用户在落地页选择套餐
    ↓
点击「立即开通」按钮
    ↓
调起内购（Google Play / Apple Store）
    ↓
支付成功回调
    ↓
开通对应云服务
    ↓
跳转到「我的服务」页面
```

### 支付环节埋点：

| 环节 | 上报事件 | 上报参数 |
|------|------|------|
| 选择套餐 | `plan_select` | 套餐类型、价格 |
| 点击开通 | `pay_click` | 套餐类型、价格 |
| 支付成功 | `pay_success` | 套餐类型、价格、订单ID |

## 八、埋点上报

### 8.1 事件列表

| 序号 | 事件名 | 触发时机 | 优先级 |
|------|------|------|------|
| 1 | `experiment_view` | 落地页首屏加载完成 | P0 |
| 2 | `plan_select` | 用户点击选择某个套餐 | P0 |
| 3 | `pay_click` | 用户点击「立即开通」或支付按钮 | P0 |
| 4 | `pay_success` | 支付成功回调 | P0 |

### 8.2 公共参数

所有事件上报时需携带以下公共参数：

| 参数 | 类型 | 说明 |
|------|------|------|
| event_type | String | 事件类型 |
| user_id | String | 用户唯一标识 |
| experiment_key | String | 实验标识（如 `ai_landing_test_01`） |
| variant | String | 变体 A / B |
| segment | String | 用户分群 |
| timestamp | Number | 事件时间戳(ms) |

### 8.3 事件参数

| 事件 | 特有参数 |
|------|------|
| `experiment_view` | page_url |
| `plan_select` | plan_type (basic/pro)、plan_price |
| `pay_click` | plan_type、plan_price |
| `pay_success` | plan_type、plan_price、order_id |

### 8.4 上报时机说明

- `experiment_view`：页面 DOM 加载完成且首屏内容可见时上报，每个用户进入页面仅上报一次
- `plan_select`：用户点击套餐选择按钮时立即上报
- `pay_click`：用户点击支付/开通按钮时立即上报
- `pay_success`：收到支付成功回调后立即上报

### 8.5 去重策略

| 场景 | 策略 |
|------|------|
| 同一用户重复曝光 | 1 分钟内同一 user_id + experiment_view 去重 |
| 支付成功重复上报 | 同一 order_id 仅保留第一条 |

## 九、实验收敛（发版清理）

实验结束后，APP 下次发版时需：

1. **移除实验分流代码**：删除请求后台分流的逻辑
2. **将胜出变体设为默认**：将胜出版本的内容固化，删除失败版本的代码/资源
3. **清理实验 Key**：移除不再使用的实验 Key 引用

```
实验运行 → 数据显著 → 运营声明胜出 → 后台全量切到胜出版本
                                              ↓
                              APP 下次发版：移除实验代码，胜出版本设为默认
```
